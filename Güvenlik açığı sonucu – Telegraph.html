Güvenlik açığı sonucu


🔒 Güvenlik Tarama Raporu – Parax (parax-vqqb.onrender.com)

🔴 Kritik Seviyede Açıklar (Yayına Almadan Önce Mutlaka Çözülmeli)

1. Sunucu Kaynak Kodları Herkese Açık

Tüm TypeScript kaynak kodları express.static() üzerinden herkese açık şekilde servis ediliyor.

Doğrudan erişilebilen bazı dosyalar:

  • /server/src/server.ts → Sunucu yapılandırması, port ve CORS ayarları
  • /server/src/routes/auth.ts → Firebase Admin SDK kimlik doğrulama işlemleri
  • /server/src/routes/voice.ts → Daily.co API kullanımı
  • /server/src/middleware/auth.ts → Token doğrulama mantığı
  • /server/src/config/firebase.ts → Firebase yapılandırması
  • /firebase.json → Firebase proje konfigürasyonu
  • /server/ ve /functions/ dizinleri

Risk: Uygulamanın iç yapısı, endpoint'ler ve güvenlik mantığı üçüncü kişilere tamamen açığa çıkıyor.

Çözüm: express.static() yalnızca public/ gibi gerekli dosyaların bulunduğu dizini servis etmeli.

2. Firestore Kuralları ile Yetki Yükseltme (Privilege Escalation)

match /servers/{serverCode}/roles/{roleId} {
  allow create, update, delete: if request.auth != null;
}

Kimliği doğrulanmış herhangi bir kullanıcı:

  • Rol oluşturabiliyor
  • Rol düzenleyebiliyor
  • Rol silebiliyor

Bu durum saldırganın kendine yönetici yetkisi vermesine ve tüm sunucular üzerinde tam yetki elde etmesine neden olabilir.

Çözüm: Rol işlemleri için manage_roles veya eşdeğer bir yetki kontrolü uygulanmalı.

3. Sunucu Üyeliği Korumasız

match /serverMembers/{docId} {
  allow create: if request.auth != null;
}

Kimliği doğrulanmış herhangi bir kullanıcı, sunucu kodunu bildiği sürece istediği sunucuya katılabiliyor.

Risk:

  • Özel sunuculara izinsiz erişim
  • Mesajların okunabilmesi
  • Üye bilgilerinin görüntülenebilmesi

Çözüm: Davet sistemi veya mevcut üye/yönetici onayı zorunlu hale getirilmeli.

🟠 Yüksek Öncelikli Açıklar

4. Oda Şifreleri Düz Metin Olarak Saklanıyor

if (password) {
  data.password = password;
}

Şifreler hash'lenmeden Firestore'a kaydediliyor.

Çözüm: bcrypt veya benzeri güvenli bir algoritma kullanılmalı.

5. Avatarlar Base64 Olarak Firestore'da Saklanıyor

Avatarlar doğrudan Firestore'a yazılıyor.

Riskler:

  • Çok yüksek okuma/yazma maliyeti
  • Performans sorunları
  • Doküman boyutu problemleri

Çözüm: Dosyalar Firebase Storage üzerinde tutulmalı.

6. Kanal Oluşturma Yetki Kontrolü Yok

Kimliği doğrulanmış herhangi bir kullanıcı istediği sunucuda kanal oluşturabiliyor.

Çözüm: manage_channels yetkisi kontrol edilmeli.

🟡 Orta Öncelikli Açıklar

7. Güvenlik Header'ları Eksik

  • ❌ Content-Security-Policy
  • ❌ X-Frame-Options
  • ❌ X-Content-Type-Options
  • ❌ Strict-Transport-Security
  • ❌ Referrer-Policy
  • ❌ Permissions-Policy

8. CORS Fazla İzin Veriyor

cors({
  origin: true,
  credentials: true
})

Her origin'e izin verilmesi güvenlik riskini artırıyor.

Çözüm: Yalnızca izin verilen domain'ler tanımlanmalı.

9. Zayıf Şifre Politikası

  • Minimum 6 karakter
  • Karmaşık şifre zorunluluğu yok
  • Rate limiting bulunmuyor

10. Math.random() ile Kod Üretimi

Math.floor(10000000000 + Math.random() * 90000000000)

Kriptografik olarak güvenli değildir.

Çözüm: crypto.randomInt() veya UUID kullanılmalı.

11. errors Koleksiyonu Korumasız

Hata logları içerisinde:

  • userId
  • URL
  • userAgent

gibi bilgiler bulunuyor ve erişim yeterince sınırlandırılmamış.

12. /api/auth/profile Endpoint'i

Kimliği doğrulanmış kullanıcılar:

  • uid
  • email
  • displayName

bilgilerini alabiliyor ve endpoint üzerinde rate limiting bulunmuyor.

🟢 Düşük Öncelikli Konular

  • package.json herkese açık.
  • Daily.co API kullanımı kaynak kod sızıntısıyla birlikte risk oluşturuyor.
  • Eski oda sistemi ve yeni sunucu sistemi aynı anda çalışıyor.

Öncelikli Yapılması Gerekenler

  1. express.static() yalnızca public/ dizinini servis etmeli.
  2. Firestore yetki kuralları yeniden yazılmalı.
  3. Sunucu üyeliği ve davet sistemi güvenli hale getirilmeli.
  4. Oda şifreleri hash'lenmeli.
  5. Avatarlar Firebase Storage'a taşınmalı.
  6. Güvenlik header'ları eklenmeli.
  7. CORS yapılandırması daraltılmalı.
  8. Rastgele kod üretimi güvenli hale getirilmeli.
  9. errors koleksiyonu erişim kuralları kısıtlanmalı.

Sonuç

Yayına alınmadan önce özellikle aşağıdaki üç açık mutlaka giderilmelidir:

  • Kaynak kodlarının herkese açık olması
  • Firestore üzerinden yetki yükseltme açığı
  • Korumasız sunucu üyelik sistemi

Bu üç açık tek başına sistemin tamamen ele geçirilmesine neden olabilecek seviyede kritiktir.
